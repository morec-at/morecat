2. Article の Event Sourcing / CQRS フロー

コマンド（書込）→ Firestore append → Eventarc 通知 → RMU が読取モデルを再構築、までの一連。outbox は使わない。

sequenceDiagram autonumber actor A as 著者 (editor) participant API as API (Cloud Run) participant FS as Firestore (events) participant EV as Eventarc participant RMU as RMU (Scala.js/Node) participant SQL as Cloud SQL (読取モデル) participant V as viewer A->>API: コマンド (Drafted/Revised/Published/Unpublished)
httpOnly Cookie Note over API,FS: 同一 Firestore Tx で append + slug 予約 API->>FS: event append (create-only=楽観ロック)
+ slugs/[slug] 予約 (create-only) alt seq 衝突 または slug 重複 FS-->>API: 競合 API-->>A: 409 Conflict else 成功 FS-->>API: ok API-->>A: 200 OK end Note over FS,EV: doc 作成を検知 FS-)EV: doc created EV-)RMU: push (at-least-once / 順不同) RMU->>FS: その articleId の全イベントを読取 RMU->>RMU: domain.js で fold（プロジェクション再構築） RMU->>SQL: upsert（lastAppliedSeq で古い適用を skip） RMU-)V: revalidate (upsert 完了後 / read-your-write)

3. 読者リクエスト（読取経路）

公開ページの表示。viewer は ISR でキャッシュし、公開時の on-demand revalidate で即時反映。

sequenceDiagram autonumber actor R as 読者 participant V as viewer (Next.js / App Hosting) participant API as API (Cloud Run) participant SQL as Cloud SQL (読取モデル) participant GCS as GCS (メディア) R->>V: GET /posts/[slug] alt ISR キャッシュ有効 V-->>R: キャッシュ済み HTML else 再生成 V->>API: GET 記事 (クエリ) API->>SQL: SELECT (プロジェクション) SQL-->>API: 記事データ API-->>V: JSON V-->>R: SSR HTML end R->>GCS: 画像取得（公開読取）
